1. Introduction

1.1 Purpose

The purpose of this document is to build an online shopping system that will use augmented reality to show the buyers a 3D view of their products.

1.2 Document Conventions

1. AR - Augmented Reality

1.3 Intended Audience and Reading Suggestions

This project is a prototype for the online augmented reality shopping system. The intended readers of this document are the developers of system and course instructor. The entire SRS is divided into six(6) sections. The first section is the Introduction. Overall project description is provided in section two. Section three provides the External Interface Requirements including user, software, hardware and communication interface. Readers will know about the System Features in section four. Other Nonfunctional Requirements are described in section five and section six is used for further more requirements. Readers are suggested to go through the SRS from section one to section six sequentially.

1.4 Product Scope

The purpose of this product is to give the customers an idea about the products they are buying. By using AR they can not only see pictures but also check if that product meet their needs. Nowadays people do not trust online shopping system because they don't get what they want. Our goal is to provide a facility that will enable them to see what customers are buying and gain back their trust. With the biggest companies in the world investing billions on Augmented Reality and AR integrations in e-commerce platforms to enhance shopping, it won't be long before the Augmented Reality retail shopping industry has over 130 Billion streaming annually. Analysts predict by 2020 AR shopping will be almost a 150 Billion dollar industry.

2.Overall Description

This section will give an overview of the whole system. The system will be explained in its context to show how the system interacts with other systems and introduce the basic functionality of it. It will also describe what type of stakeholders that will use the system and what functionality is available for each type. At last, the constraints and assumptions for the system will be presented.

2.1 Product Perspective

This system will consist of two parts: one android application and one web portal. The mobile application will be used by the buyers who will be able to buy the products and view the products in AR. The web portal will be used by both the buyers and the sellers. The buyers can buy and the sellers can upload their product descriptions.
The android application needs and version 7.0 or above. Android phones with version 7.0 or above comes with necessary hardware required for augmented reality support. The mobile app will be using the camera of the smartphone to implement the AR features.
The mobile application and the web application will both have a database to store product data and user data. The web portal will have read, write and modify access of the database but the mobile app will only have read access. All database communication will happen through internet.

2.2 Product Functions

With the mobile application the user will be able to buy furniture. The special feature of this app will be to see the product before buying in Augmented Reality. The users will also be able to search for a specific product with name or find it from the category menu.
The web portal will have two parts, one for the administrator and another one is for the buyers. The buyers will be able to buy furniture from here also. But the website won’t be supporting any AR feature. As a result, the web portal will be accessible from any device as it won’t have any hardware constrain. The admins of the portal will be able to add new products with details, receive orders and also get analytical data on sales.

2.3 User Classes and Characteristics

There are two types of users of the platform. The buyers and the administrators.
The buyers will be able to use the mobile app and the web app. They can use all the features the mobile app provides.
The administrator will take product data from the sellers and will upload them on the web portal. They will not have access to any management features from the mobile application.

2.4 Operating Environment

For Android:

Operating System: Android 7.0
Disk Space: 100 MB
RAM: 1 GB

2.5 Design and Implementation Constraints

As this an online application, internet connection is a constraint. Internet connection is crucial for the application to fetch data from the database over the internet.
Both the web portal and the mobile application will be constrained by the capacity of the database. Since the database is shared between both application it may be forced to queue incoming requests and therefor increase the time it takes to fetch data.
Android version is a mobile software constraint as the SDK used requires android version 7.0 and above. Also, the need of a camera, gyro, accelerometer is hardware constraint for the android application as without these the AR feature will not work.

2.6 User Documentation

The platform is just entering development stage. Once the development is completed and a prototype a released user documentation will be provided.

2.7 Assumptions and Dependencies

One assumption for the android application is the application will be used mostly on high-end or latest phones as for a few years as the hardware requirement can only be met on those phones.
Another assumption is phones with good camera will be providing the best performance as plane detection depends on the picture quality of the camera.
Proper browser should be used for the web app and a stable internet connection is required.

3.2 Hardware interfaces

The most important hardware Shopline requires is a smartphone with a built in camera. The device should be able to activate the camera from within the app to make the sceneform 3D view.The secondary requirement is an active Internet connection, either by mobile data or WiFi.This is to communicate with the Google Firebase Server and to make API calls to the google Sceneform

3.3 Software interfaces

Shopline app will run Android 7.0 or later devies which suppors Google Play Services for AR, while a browser-based web interface will run on any computer (Windows, Mac or Linux) with a capable modern web browser. Google’s ArCore Platform will be used to take the advantage of its Sceform based 3D augmented reality functions.We will be using the free versions of the products,for this project purposes.

3.4 Communication interfaces

The application will set 3D image to the google ARCore Sceneform.ARCore will do two things: tracking the position of the mobile device as it moves, and building its own understanding of the real world.ARCore's motion tracking technology uses the phone's camera to identify interesting points, called features, and tracks how those points move over time. With a combination of the movement of these points and readings from the phone's inertial sensors, ARCore determines both the position and orientation of the phone as it moves through space.

5.Other Non-functional Requirements

5.1 Performance Requirements

The system will be able to accommodate a large number of items and users. Also, in every action-response of the system, the delay will not be longer than a few seconds.

5.2 Safety Requirements

All the data users' need to store, for the future, is always safe with our cloud storage. Even if the user device is broken,through his/her account the data will always remain protected.

5.3 Security Requirements

The system will use a secure database. Users can go through basic information but will not be able to modify any information except their personal ones. Also, each user of the system shall login using some form of unique identification (e.g., username and password). All login attempts shall be done so in a secure manner (e.g., encrypted passwords)

5.4 Software Quality and Attributes

5.4.1 Availability: The system shall be available at all times, that is, the user can access it using a web browser or mobile app, only restricted by the down time of the server on which the system runs. Also in case of a hardware failure, backups of the database can be retrieved from the server and saved by the administrator. Then the service will be restarted.

5.4.2 Portability: The end-user part is fully portable and any system using any web browser should be able to use the features of our system. The web part of the system can be used on Windows or Mac etc and the mobile app version can run on Android.

5.4.3 Usability: The system is easy to handle and navigates without much delay, and so, the system program reacts accordingly and transverses quickly between its states.

5.4.4 GUI: The system shall provide a uniform look and feel between all the web pages, along with digital image for each of the products.

5.5 Business Rules

The system requires the user to have an account (could be accessed via facebook or google account as well) in order to shop. The user has to be logged in as well. The system will have the option for the users to pay upon delivery of the item.

6.Appendix

6.1 Glossary

1. AR - Augmented Reality. It is the integration of digital information with the user's environment in real time.

3. GUI - Graphical User Interface. It is a system of interactive components such as icons and other graphical objects that help a user interact with computer software.

4. SDK - Software Development Kit. It is a collection of software development tools in one installable package.
